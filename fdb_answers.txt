1. TMS is made up of a Sequencer, Proxies, and Resolvers that work together to handle transaction processing for FDB. The Storage System handles data storage and read serving. These two layers are decoupled because it allows FDB to quickly recover from failure, and each layer can scale independently.
2. The Resolvers are the components responsible for detecting read-write conflicts. This occurs when a read range is sent to the Resolvers, where they compare the range against other transactions in the current range. If the transactions do not overlap, the Resolvers allow the transaction to commit.
3. The read version acts as a snapshot of the database at the time of the read. This ensures isolation of the transaction, so even if the committing of the transaction occurs later, it will still be as if the database acted at the time the transaction started. It also helps to detect if a conflict occurs.
4. Snapshot reads don't add the keys to the conflict set of the transaction, meaning that they will not abort the transaction if it causes a conflict. You might use them when you do not want to risk a conflict; they can improve performance when reading a hotspot key, as long as the transactions are being performed safely.
5. StorageServers continuously pull data from the LogServers, ensuring they are up to date at almost all times. Since they are decoupled, when there is a failure, the system can reboot the new LogServer with a small number of recent transactions to quickly catch up.
